# Glossary

The agentic harness space moves fast and terminology is inconsistent. This glossary defines the terms as used in this repo and across the industry.

---

### Agent
A system where an LLM autonomously decides which actions to take to accomplish a task, using tools to interact with the world and iterating until the task is done or a limit is hit.

### Agent Loop
The core execution cycle: send context to the LLM, parse the response, execute any tool calls, append results to history, and repeat. Also called the **observe-choose-act cycle** or **ReAct loop**.

### Agent-Computer Interface (ACI)
Term coined by the SWE-agent team. The set of tools and their interfaces that an agent uses to interact with the environment. Better ACI design (clear feedback, bounded output, guardrails) dramatically improves task completion.

### Agent SDK
A library that provides a complete agent runtime (loop, tools, context management, error recovery) as importable code. Examples: Anthropic's Agent SDK (Claude Code's engine as a library), OpenAI's Agents SDK.

### Agentic Harness
The complete runtime system that sits between an LLM and the real world. Includes the agent loop, tool system, context management, model routing, permissions, and UX. Distinct from the model itself — the harness determines how effectively the model's capabilities are used.

### Background Agent
An agent that runs asynchronously without continuous user supervision. The user kicks off a task and the agent reports back when done. Examples: Cursor background agents, Devin, GitHub Copilot Workspace.

### Compaction
Reducing the size of conversation history by summarizing older messages while preserving recent ones. Triggered when context usage exceeds a threshold (typically 60%). Essential for long-running sessions.

### Context Window
The maximum number of tokens an LLM can process in a single request. Harnesses must manage conversation history to stay within this limit. As of 2026, ranges from 8K (small local models) to 1M+ (Claude).

### Diff-Based Editing
A tool design pattern where the agent specifies file changes as search-and-replace operations rather than rewriting entire files. Reduces token usage and error rate. Used by Claude Code (Edit tool), Aider, and Cline.

### Eval / Evaluation
Measuring harness quality using standardized task suites. Separates model quality from harness quality — the same model produces different results in different harnesses.

### Fluid Compute
Vercel's compute model that reuses function instances across concurrent requests, reducing cold starts. Relevant for agent harnesses deployed as web services.

### Guardrail
An automatic check that runs after an agent action to catch errors. Example: lint-gating edits (running a linter after every file edit and reverting if errors are found).

### Hook
A programmatic interception point in the agent loop. **Pre-tool hooks** run before tool execution (for validation, blocking). **Post-tool hooks** run after (for formatting, logging). **Stop hooks** run when a session ends.

### Lint Gate
A specific guardrail that runs a linter/formatter after file edits. If the edit introduces syntax errors, it's reverted. Pioneered by SWE-agent.

### MCP (Model Context Protocol)
An open protocol (created by Anthropic) that standardizes how LLMs connect to external tools and data sources. Enables a plugin ecosystem where tools are defined once and work across multiple harnesses.

### Model Routing
Selecting different LLMs for different tasks based on complexity. Simple tasks go to cheap/fast models (haiku), complex tasks go to powerful models (opus). Saves 50-80% on costs.

### Multi-Agent / Sub-Agent
Spawning child agents to handle subtasks in parallel. The parent agent delegates, collects results, and synthesizes. Sub-agents typically have restricted tool access and fresh context.

### Permission System
Controls what tools an agent can use and under what conditions. Models: **allowlists** (specify allowed tools), **deny/ask/allow** (three-tier per tool), **hooks** (programmatic interception).

### Prompt Caching
An optimization where the LLM provider caches the stable prefix of a prompt (system instructions + tool schemas) and only processes new tokens. Saves 50-90% on input token costs for multi-turn sessions.

### Provider Abstraction
A gateway layer that normalizes different LLM APIs (OpenAI, Anthropic, local models) behind a single interface. Enables multi-model strategies without changing application code.

### Repo Map
A compact representation of a codebase generated by parsing code with tree-sitter. Shows file structure, class names, and function signatures without including full source code. Pioneered by Aider.

### Sandbox
An isolated execution environment that restricts what an agent can do at the system level. Types: **OS-native** (Seatbelt/Landlock — kernel enforcement), **container** (Docker), **VM** (cloud microVMs).

### Streaming
Sending LLM output to the user token-by-token as it's generated, rather than waiting for the complete response. Critical for UX — users who see progress tolerate longer tasks.

### SWE-bench
The standard benchmark for coding agents. 2,294 real GitHub issues from Python repos. **SWE-bench Verified** is a curated 500-task subset with human-verified solutions.

### Tool
A function the LLM can call to interact with the world — read a file, run a command, search the web. Defined as a JSON schema (name, description, input parameters) and executed by the harness.

### Tool Call
A structured request from the LLM to execute a specific tool with specific arguments. The harness parses tool calls from the LLM response, executes them, and feeds results back.

### Turn
One complete cycle of the agent loop: LLM receives context, generates a response, tool calls are executed, results are appended. A session typically involves 5-50 turns.

### Working Memory
A distilled representation of the current task state (files changed, decisions made, errors encountered) that's always kept in context. Distinguished from the full transcript, which may be compacted or stored externally.

### Worktree
A git feature that creates an independent working copy of a repository. Used by some harnesses to give sub-agents an isolated copy of the codebase where they can make changes without affecting the main working directory.
